# claude-memory-setup-prompt

프로젝트에서 클로드 사용시 바로 복사해 사용할 수 있는 초기 메모리 세팅 프롬프트입니다.

이 프롬프트는 프로젝트 루트에 `CLAUDE.md`를 생성하고, `.claude/memory/` 기반의 프로젝트 메모리 구조를 초기화하도록 클로드에게 지시합니다.

## 목적

클로드가 프로젝트를 진행하면서 파악한 정보를 일회성으로 흘려보내지 않고, 이후 작업에서도 참고할 수 있도록 프로젝트별 컨텍스트와 메모리를 관리하게 만드는 것이 목적입니다.

예를 들어 다음과 같은 정보를 계속 누적할 수 있습니다.

- 프로젝트 기술 스택
- 디렉토리 구조
- 주요 설계 결정
- 개발 규칙
- 개발 히스토리
- 개발 가이드
- 기획/설계 문서
- 클로드가 작업 중 파악한 프로젝트 맥락

## 특징

단순히 초기 문서를 생성하는 프롬프트가 아닙니다.

클로드가 프로젝트를 진행하면서 새롭게 파악한 정보가 이후 작업에 참고될 수 있다고 판단되면, 별도의 지시가 없어도 `CLAUDE.md`와 메모리 파일을 자율적으로 업데이트하도록 규칙을 설정합니다.

또한 메모리 파일이 늘어나거나 내용이 복잡해질 경우, Claude가 `.claude/memory/` 내부의 문서 구조를 필요에 따라 재구성할 수 있도록 합니다.

기본적으로 Claude의 프로젝트 메모리는 `~/.claude` 하위에 저장되기 때문에 개발자가 현재 메모리 상태를 직접 확인하기 어렵습니다.

이 프롬프트는 메모리 인덱스를 프로젝트 내부의 `.claude/memory/MEMORY.md`에도 동기화하도록 설정합니다.

덕분에 개발자는 클로드가 현재 어떤 정보를 기억하고 있는지, 어떤 문서를 참고 대상으로 삼고 있는지 프로젝트 디렉토리 안에서 바로 확인할 수 있습니다.

## 메모리 구조

모든 프로젝트 지식 파일은 `.claude/memory/` 아래에 통합됩니다.

```text
project-root/
├── CLAUDE.md
└── .claude/
    └── memory/
        ├── MEMORY.md
        └── ...
```

`MEMORY.md`는 메모리 파일들의 인덱스 역할을 합니다.

이 인덱스는 아래 두 경로에 동기화됩니다.

```text
~/.claude/projects/<project-path>/memory/MEMORY.md
project-root/.claude/memory/MEMORY.md
```

클로드는 `MEMORY.md`를 수정할 때마다 두 경로를 함께 업데이트하도록 지시받습니다.

이 구조를 통해 클로드 내부 메모리와 프로젝트 내부 메모리 인덱스가 어긋나지 않도록 유지할 수 있습니다.

개발자는 프로젝트 루트의 `.claude/memory/MEMORY.md`를 열어 클로드의 메모리 상태를 바로 확인할 수 있습니다.

## 사용법

클로드 세션에서 프로젝트를 연 직후, 최초 컨텍스트 셋업 시점에 **`PROMPT.md`의 내용을 그대로 복사해서 클로드에게 전달합니다.**

최초 프롬프트 전달 이후 새로운 클로드 세션에서 대화를 시작해도 일관적인 동작을 유지합니다.

## 멀티 레포 프로젝트에서의 사용

프론트엔드, 백엔드, 여러 서비스가 하나의 프로젝트를 구성하는 멀티 레포 구조라면 각 레포에서 따로 클로드 세션을 시작하기보다, 관련 레포들이 모여 있는 부모 디렉토리에서 클로드 세션을 시작하는 것을 권장합니다.

예를 들어 아래와 같은 구조라면:

```text
Project/
├── Project-fe/
├── Project-be-service-a/
├── Project-be-service-b/
└── Project-be-service-c/
```

`Project-fe`, `Project-be-service-a` 각각에서 따로 초기 세팅을 하기보다 `Project/` 디렉토리에서 Claude 세션을 열고 초기 세팅을 진행하는 방식이 더 적합합니다.

이렇게 하면 클로드가 개별 레포만이 아니라 전체 프로젝트 구조를 기준으로 컨텍스트를 관리할 수 있습니다.

예를 들어 다음과 같은 정보를 하나의 메모리 시스템 안에서 함께 관리할 수 있습니다.

- 프론트엔드와 백엔드의 연결 구조
- 서비스 간 의존 관계
- 공통 개발 규칙
- API 연동 흐름
- 배포/운영 구조
- 여러 레포에 걸친 설계 결정과 개발 히스토리

즉, 여러 레포가 하나의 제품이나 시스템을 구성한다면 가장 상위의 프로젝트 디렉토리에서 이 프롬프트를 적용하는 것을 추천합니다.

## 기대 효과

- 프로젝트별 컨텍스트 유지
- 반복 설명 감소
- Claude의 작업 일관성 향상
- 설계 결정과 개발 히스토리 누적
- Claude 메모리 상태를 프로젝트 내부에서 바로 확인
- 장기 프로젝트에서의 메모리 손실 방지
